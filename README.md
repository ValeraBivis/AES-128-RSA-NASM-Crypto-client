# AES-128-RSA-NASM-Crypto-client
**Гибридный крипто-клиент AES-128 / RSA на NASM**

**ЕСЛИ У ВАС НЕ РАБОТАЮТ КОМЕНТАРИИ ИЛИ РУССКИЙ ЯЗЫК В КОНСОЛИ СЛОМАН, СМЕНИТЕ КОДИРОВКУ НА OEM 866**

Создан с целью проверить, насколько реально с нуля написать что-то подобное и возможно ли это для кого-то не столь опытного? 

**Принцип работы**

 1.   **Ввод данных**: Запрашиваем у пользователя абсолютный путь к файлу (или название, если файл лежит в одной папке с клиентом).  

2.    **Выравнивание (Паддинг)**: Для работы блочных шифров необходимо использовать выравнивание данных, чтобы размер файла был строго кратен 16 байтам (длине ключа AES-128).  

3.    **Генерация ключа AES**: На данном этапе для генерации ключа используются процессорные инструкции rdtsc и cpuid. Для обеспечения криптографической стойкости в будущих релизах планируется переход на системный метод BCryptGenRandom.  

4.    **Расширение ключа**: С помощью аппаратных инструкций процессора AES-NI разворачиваем один исходный 16-байтный ключ в 11 раундовых ключей (всего 176 байт памяти).  

5.    **Шифрование файла (AES-128-CBC)**: В текущей версии из-за ограничения буфера в 4 КБ обрабатываются относительно небольшие файлы (в будущих версиях размер буфера будет увеличен). Используем вектор инициализации IV (сейчас он статично зашит в коде, в релизе v2.0 это будет исправлено). Данные шифруются блоками по 16 байт и сцепляются через операцию XOR (pxor). Чтобы исключить трансформацию MixColumns на финальном 10-м раунде, применяется инструкция aesenclast. Результат работы сохраняется в локальный файл encrypted.bin.  

6.    **Инициализация сети**: Поднимается сетевая подсистема Windows (Winsock), и сокет подключается к серверу. (В текущей сборке адрес жестко задан как localhost / 127.0.0.1, в следующей версии будет добавлен динамический ввод IP и порта через консоль).  

7.    **Обмен ключами и верификация (Защита от MITM)**: Принимаем от сервера-дешифратора блоб его публичного RSA-ключа. Чтобы исключить атаку «человек посередине» (MITM), передаем этот блоб криптопровайдеру Windows (CNG API), вычисляем его SHA-256 отпечаток, преобразуем первые 4 байта хэша в читаемый ASCII-вид (8 символов формата XX-XX-XX-XX) и выводим на экран. Программа приостанавливает работу и ждет от пользователя ввода yes для подтверждения (принцип заимствован из механики верификации хостов в SSH).  

8.    **Шифрование и отправка сессионного ключа**: Если пользователь подтвердил отпечаток, сырой блоб парсится в системный хэндл RSA-ключа, после чего наш секретный AES-ключ шифруется методом RSA с паддингом PKCS#1 v1.5 и отправляется в сокет. Следом за ним улетает зашифрованный файл.  

 9.   **Очистка ресурсов**: После завершения передачи закрываем сокеты и хэндлы файлов. По задумке, сессионный AES-ключ полностью уничтожается в памяти после закрытия процесса, что исключает его повторное использование — количество шифровок должно строго совпадать с количеством расшифровок.



По моим планам, после завершения основных правок, перехват файла и даже зашифрованного ключа AES в сети не даст атакующему ничего: расшифровать их будет математически невозможно. Единственным вектором атаки останется компрометация конечных ПК (отправителя или получателя) либо социальная инженерия.
